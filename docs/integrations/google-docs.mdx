# Google Docs Integration via OpenInt

Connect, read, and (limited) interact with Google Docs from your application using OpenInt. This guide covers step-by-step configuration for adding OAuth-based Google Docs integration, key limitations, and troubleshooting for engineering teams embedding OpenInt.

> ℹ️ **Status**: This connector is **general availability (GA)**, but only supports access to the _current Google Docs document_ via OAuth scope `documents.currentonly`. Broader features (such as listing or editing all Docs) require broader scopes, not yet available via OpenInt.

---

## Prerequisites

- Google Account
- Access to [Google Cloud Console](https://console.developers.google.com/)
- OpenInt dashboard/API access

---

## Step 1: Create OAuth Credentials in Google Cloud

To interact with a user's Google Docs via OAuth2, your integration needs a valid OAuth **Client ID** and **Client Secret**. Follow this process (adapted from [2BrightSparks guide](https://www.2brightsparks.com/resources/articles/gdriveclientid.html)):

<Steps>
  <Step title="Select or Create your Google Cloud Project">
    Go to [Google API Console](https://console.developers.google.com/) and sign in. Use the project selector—create a _new project_ if needed. Name it clearly (e.g., "Company Integration").
  </Step>

  <Step title="Enable Google Docs and Google Drive APIs">
    In the sidebar, click <b>APIs & Services > Enabled APIs & services</b>. Use "+ ENABLE APIS AND SERVICES" to search for and enable 
- <b>Google Docs API</b>
- <b>Google Drive API</b> (required for scope compatibility)
  </Step>
  
  <Step title="Configure the OAuth Consent Screen">
    Go to <b>OAuth consent screen</b>. For most use-cases select <b>External</b>, then click <b>Create</b>.<br/>
    Fill out <b>App name</b>, <b>User support email</b>, and <b>Developer contact info</b>. Follow prompts, save.
    Add test users (your developer email) for non-production or testing environments.
  </Step>
  
  <Step title="Create OAuth Client Credentials">
    Go to <b>Credentials</b> in the sidebar.<br/>
    Click <b>+ CREATE CREDENTIALS</b> → <b>OAuth client ID</b>.
    Choose <b>Web application</b> (for web apps), or <b>Desktop app</b> as appropriate.
    - Set the correct <b>Authorized redirect URIs</b> <br/>(see your OpenInt dashboard for the correct redirect URI)
    - Name the credentials (e.g. "OpenInt Google Docs Connector")
    Save: you will be shown your <b>Client ID</b> and <b>Client Secret</b> — copy these securely.
  </Step>

  <Step title="(Optional) Publish or Prepare for Verification">
    If you plan to go live to external users, return to the <b>OAuth consent screen</b> and click <b>Publish app</b>.<br/>
    <Callout type="warning">Do not initiate "Prepare for Verification" unless you require broader scopes and intend public launch—Google reviews are strict and can take time.</Callout>
  </Step>
</Steps>

## Step 2: Configure OpenInt

Provide your Google OAuth2 **Client ID** and **Client Secret** in your OpenInt dashboard or configuration as required.

<CodeGroup>
```json OpenInt Connector Config
{
  "provider": "google-docs",
  "client_id": "YOUR_GOOGLE_CLIENT_ID",
  "client_secret": "YOUR_GOOGLE_CLIENT_SECRET",
  "scopes": [
    "https://www.googleapis.com/auth/documents.currentonly"
  ]
}
```
</CodeGroup>

- <b>Supported scope:</b> Only `https://www.googleapis.com/auth/documents.currentonly` is supported at this time. This scope grants access ONLY to the currently open Google Document (typically via a Docs add-on or specific user gesture):
  - Most restrictive; does <i>not</i> allow listing, editing, or accessing other Docs or Drive files.
- For product roadmap or broader access, contact OpenInt support or track this page.

---

## Step 3: Authentication Flow

1. OpenInt will generate an OAuth2 authorization link for the user.
2. User consents, authenticates via Google’s prompt ("consent" always set).
3. Google redirects back to OpenInt's configured redirect URI with an auth code.
4. OpenInt exchanges the code for an access (and if needed, refresh) token for just the allowed scope.
5. You may now use the OpenInt API or SDK to interact (in a limited way) with the authorized Google Doc session.

## Common Pitfalls & Troubleshooting

- <b>Redirect URI mismatch:</b> Ensure your Google Cloud Console OAuth client’s redirect URI <i>exactly</i> matches the one OpenInt provides in settings.
- <b>Wrong scopes:</b> If you request any scope except `documents.currentonly`, authorization will fail.
- <b>Unverified app:</b> If your consent screen is not published, only test users can proceed. For production, you may require verification; follow <a href="https://developers.google.com/workspace/guides/verification">Google OAuth verification guide</a>.
- <b>Client Secret in frontend:</b> Never embed secrets in client-side code—handle tokens on the server or use OpenInt’s backend integration.
- <b>Consent screen warnings:</b> Warnings about unverified scopes are expected if using anything but the minimal scope—stick with supported scopes while in OpenInt.
